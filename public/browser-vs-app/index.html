Feel & feedback
Vibration & haptics
button taps, score pulses, collision rumble
 No
Apple blocks this entirely in the browser.
navigator.vibrate() not implemented in WebKit. No path to Taptic Engine from the browser. No workaround exists.
 Basic only
Android phones can vibrate from a browser, but it is blunt: on/off and simple patterns only. Requires a user gesture (tap or click) to trigger.
navigator.vibrate() supported — on/off and pattern arrays. No amplitude control, no frequency shaping, no semantic effect types.
 Full
Full access to the Taptic Engine. Every tap, click, and pulse style. Can sync with audio.
Core Haptics (CHHapticEngine, iOS 13+): transients, continuous events, intensity/sharpness 0–1, parameter curves, AHAP patterns, synchronized audio+haptics.
 Full
Full control over strength, rhythm, and type. Varies by device hardware.
VibrationEffect with amplitude control (API 26+); VibrationEffect.Composition with named primitives — PRIMITIVE_CLICK, PRIMITIVE_LOW_TICK, etc. (API 30+).

hiding browser chrome, immersive game UI
 Not on iPhone
Browser pages on iPhone cannot go edge-to-edge. The status bar is always visible. Works on iPad.
Element.requestFullscreen() unsupported on iPhone. Video-only via webkitEnterFullscreen(). Status bar, Dynamic Island, home indicator always visible. iPad: full API since Safari 16.4.
 Yes
Full screen works. Status bar and navigation bar both disappear. Must be triggered by a user gesture (tap or click).
Element.requestFullscreen() fully supported (Chrome 71+, unprefixed). PWA with display:fullscreen hides all system chrome.
 Yes
Full edge-to-edge. Status bar and home indicator can be hidden.
prefersStatusBarHidden + prefersHomeIndicatorAutoHidden.
 Yes
Full edge-to-edge. All system bars hidden.
WindowInsetsController.hide(WindowInsets.Type.systemBars()).

Sensors
Tilt & rotation
steering, aiming, physical gesture controls
 Drifts over time
Tilt and rotation work, but compass direction drifts over time without correction. Fine for simple tilt controls, not reliable for pointing or aiming. Requires a permission prompt that can only be shown after a user gesture (tap or click). If the user denies it, you cannot ask again.
DeviceOrientationEvent: accel + gyro only. Alpha (yaw) is relative, no magnetometer fusion, drifts over time. absolute always false. Requires DeviceOrientationEvent.requestPermission() (iOS 13+); if denied, origin cannot re-prompt.
 Accurate and stable
Android browsers combine accelerometer, gyroscope, and compass into a stable, drift-free reading. Direction stays accurate. No permission prompt needed.
Generic Sensor API (Chrome 67+): AbsoluteOrientationSensor fuses accel + gyro + magnetometer into drift-corrected quaternion. No permission prompt. ~60 Hz cap.
 Precise and fast
All sensors fused together. Updates 100+ times per second.
CMDeviceMotion (CoreMotion): full AHRS, quaternion output, 4 reference frames, bias-removed rotation rate, 100+ Hz configurable.
 Precise and fast
Same as iPhone app. Updates hundreds of times per second on high-end hardware.
TYPE_ROTATION_VECTOR (full fusion), TYPE_GAME_ROTATION_VECTOR, quaternion output, up to 200–500+ Hz. HIGH_SAMPLING_RATE_SENSORS permission above 200 Hz (Android 12+).
Compass
directional gameplay, north-facing mechanics
 Direction only
Compass direction only. Requires a permission prompt that can only be shown after a user gesture (tap or click).
webkitCompassHeading: bearing 0–360° relative to magnetic north. No raw x/y/z data. Requires DeviceOrientationEvent permission.
 Direction works; raw data needs enabling
Compass direction works by default. Raw magnetic field data requires a non-standard browser setting most users will not have.
deviceorientationabsolute provides absolute heading by default. Raw Magnetometer API disabled by default; requires chrome://flags/#enable-generic-sensor-extra-classes. Capped at 10 Hz when enabled.
 Full
Full compass and magnetic field data.
CMMotionManager.magnetometerData (raw µT), CMDeviceMotion.magneticField (calibrated), CLLocationManager.heading.
 Full
Full magnetic field data. No permission needed.
TYPE_MAGNETIC_FIELD and TYPE_MAGNETIC_FIELD_UNCALIBRATED (API 18+). No runtime permission.
Sensor speed
fast motion tracking, low-latency response
 60 updates/sec — fixed
60 updates per second, fixed. Fine for most interactions. Sluggish for fast motion or precision mechanics.
DeviceMotionEvent fixed at ~60 Hz. Not configurable.
 60 updates/sec — fixed
Same 60/sec cap. The compass only updates 10 times per second.
Generic Sensor API accepts frequency parameter but caps at ~60 Hz. Magnetometer hard-capped at 10 Hz.
 100–200+ per sec
2-3x faster than any browser.
Configurable via deviceMotionUpdateInterval. Hardware supports 100+ Hz; some devices reach 200 Hz+.
 Up to 500 per sec
Fastest available. Well beyond what any browser can offer.
SENSOR_DELAY_FASTEST: 200–500+ Hz on flagships. HIGH_SAMPLING_RATE_SENSORS install-time permission above 200 Hz (Android 12+).
Light sensor
adapting UI to room brightness
 No
Not accessible from any browser on iPhone.
AmbientLightSensor (Generic Sensor API) not implemented in WebKit.
 Not in practice
Technically possible but requires a non-standard browser setting. Not reliable with general audiences.
AmbientLightSensor behind chrome://flags/#enable-generic-sensor-extra-classes. Disabled by default. 10 Hz when enabled.
 Not available
Apple does not expose the light sensor to apps either.
No public API. SRAmbientLightSensor (SensorKit) is research-only with months-long Apple approval process.
 Yes
Available to apps in lux. No permission needed.
TYPE_LIGHT via SensorManager. Illuminance in lux. No runtime permission.

 No

No web API exists for barometric pressure on any browser or platform.
 Yes

CMAltimeter: pressure in kPa, relative altitude. NSMotionUsageDescription + authorization required since iOS 17.4.
 Yes

Sensor latency
motion controls, real-time input, timing-sensitive mechanics
 High
Sensor events go through the browser’s main thread and are subject to scheduling delays. Expect 50–100ms of latency in practice. Fine for slow gestures, noticeable for anything timing-sensitive.
DeviceMotionEvent delivered via main thread event loop. Subject to JS scheduling jitter and browser throttling. Typical observed latency 50–100ms. No way to request lower latency or higher priority from web.
 High
Same main-thread bottleneck as iPhone browser. Chrome’s Generic Sensor API has slightly better architecture but is still meaningfully slower than native.
Generic Sensor API uses a dedicated sensor task runner but data still crosses to the main thread before JS callbacks fire. Latency improved over DeviceMotionEvent but still 30–80ms in practice.
 Low
Reads directly from hardware with no browser in the way. Typical latency under 5ms. The difference is perceptible for fast motion or precise timing.
CoreMotion delivers data on a dedicated background thread at hardware interrupt frequency. Typical latency 1–5ms. No main-thread involvement until the app chooses to process.
 Low
Same as iPhone app. Direct hardware access, sub-5ms latency.
SensorManager callbacks on a dedicated SensorThread. Typical latency 1–5ms at SENSOR_DELAY_FASTEST. No main-thread bottleneck.
Camera, flashlight & AR
Camera
face detection, QR codes, heart rate via PPG, scanning
 Works, with caveats
Works, but Safari sometimes forgets permission and re-prompts next session. The green status bar dot is OS-enforced and appears in apps too.
getUserMedia() supported. HTTPS required. Green status bar dot plus Safari-specific browser indicator. Permissions less persistent than Chrome — may re-prompt across sessions or SPA route transitions. ImageCapture API not supported.
 Works, with caveats
Works. Permission is remembered per site, so less re-prompting than iPhone.
getUserMedia() supported. HTTPS required. Green dot (Android 12+). Permissions persist per site. ImageCapture API supported in Chrome.
 Full control
Full control over resolution, frame rate, focus, and exposure. Green dot is always shown by the OS.
AVCaptureSession: full resolution, framerate, focus, exposure control. Green status bar dot OS-enforced.
 Full control
Same as iPhone app.
Camera2 / CameraX: full control over all camera parameters.
Flashlight
low-light signaling, attention mechanics
 Not reliably
Not reliably possible. Requires activating the camera at the same time, and even then it is inconsistent across devices.
ImageCapture API unsupported in Safari. Torch via applyConstraints({torch:true}) on getUserMedia track is newly possible in some Safari versions but unreliable. Always requires active camera stream.
 Yes, but camera turns on too
Works, but the camera must be active simultaneously, so the green privacy dot appears.
applyConstraints({advanced:[{torch:true}]}) on getUserMedia video track (Chrome 59+). Camera stream must be active; green dot appears.
 Independently
On/off with brightness control, without the camera running. Camera permission required but the camera itself stays off.
AVCaptureDevice.torchMode + setTorchModeOn(level:) 0.0–1.0. Camera permission required; camera session not required to be active.
 Independently
Can control the flashlight without activating the camera. No green privacy dot.
CameraManager.setTorchMode(cameraId, enabled) (API 23+). Can enable torch without opening a camera session.
Augmented reality
overlaying graphics on the real world, spatial experiences
 Limited AR
No WebXR, so no surface detection or world tracking. Marker-based and image-based AR work via JS computer vision libraries (MindAR, AR.js). 8th Wall shut down in early 2026.
No WebXR support in Safari (any version). AR Quick Look (USDZ via rel="ar") is a non-programmable viewer. getUserMedia() + JS/WASM CV works for marker tracking (AR.js) and image tracking (MindAR). No plane detection or world tracking. 8th Wall shut down Feb 28, 2026.
 Moderate AR
WebXR provides surface detection, depth sensing, and light estimation. Marker and image tracking also work via JS libraries (MindAR, AR.js), and typically run faster on Android Chrome than iOS Safari.
WebXR Device API (Chrome 79+) via ARCore: hit-test, anchors, plane detection, DOM overlay, light estimation, depth sensing. No WebXR image tracking, cloud anchors, or Geospatial API. JS/WASM CV libraries (MindAR, AR.js, OpenCV.js) provide marker and image tracking, generally at higher FPS than iOS Safari.
 Full AR
Face and body tracking, object and image recognition, LiDAR room scanning, people occlusion.
ARKit: world/face/body tracking, LiDAR scene reconstruction, image/object detection, people occlusion, location anchors, 4K/HDR AR video.
 Full AR
Full AR including geospatial anchoring to real-world locations. Multiple users can share the same AR space.
ARCore: motion tracking, ML depth, augmented faces/images, Geospatial API with VPS + Streetscape Geometry, scene semantics, cloud anchors.
Face, hand & body tracking
gesture controls, face filters, conductor input, physical performance
 Possible, but slower
Works via AI models running in the page (TensorFlow.js/MediaPipe). Covers face mesh, hand landmarks, and body pose. Frame rates are lower and battery drain is higher than a native app.
MediaPipe (WebAssembly + WebGL): Face Mesh (468 points), Hands (21 landmarks/hand), Pose (33 landmarks). ~20–30 FPS. No Neural Engine access.
 Possible, better than iPhone browser
Same libraries as iPhone browser, but tends to run faster and more efficiently on Android Chrome.
Same MediaPipe stack. Typically higher FPS than iOS Safari due to stronger WebGL/WASM optimization in Chrome.
 Fast and precise
Hardware-accelerated, built into the OS. Face mesh, 21-point hand tracking, full body skeleton. About 10ms response time.
Vision framework: VNDetectFaceLandmarksRequest (76 landmarks), VNDetectHumanHandPoseRequest (21 landmarks, 2 hands), VNDetectHumanBodyPoseRequest (19 joints). ARKit face/body tracking on A12+. ~5–14 ms latency.
 Fast and precise
Uses the phone's AI chip. Face mesh, hand, and body tracking at 2-3x faster than the browser equivalent.
ML Kit Face Mesh Detection (468 landmarks), Pose Detection (33 landmarks), GPU/NNAPI acceleration. ~2–3× lower latency than browser implementations.
Audio
Playing sound automatically
ambient music on load, sound effects timed to show events
 Can’t start without a tap
No audio until the user taps something first. Cannot start audio on page load or on a timer.
WebKit autoplay policy: any audible media (HTMLMediaElement, AudioContext) blocked until user gesture. AudioContext starts suspended; must resume() after interaction. No per-site exemptions. Applies equally to iOS PWAs.
 Needs a tap first — unless installed
Same restriction in the browser tab. If installed to the home screen, the requirement goes away.
Chrome autoplay policy: same gesture requirement in browser tab. Installed Android PWA exempt from autoplay restriction since Chrome 71.
 Play whenever
No restrictions. Sound plays when you tell it to.
AVAudioSession with .playback category. No gesture requirement.
 Play whenever
No restrictions. Sound plays when you tell it to.
MediaPlayer, ExoPlayer, AudioTrack — no autoplay restrictions.
Speaker vs. earpiece routing
private audio to one person, directional sound design
 Can’t choose
The browser decides where audio comes out, and sometimes gets it wrong. If the camera is active (e.g. for heart rate tracking), audio often routes to the earpiece with no way to override.
setSinkId() unreliable/unsupported on iOS Safari. getUserMedia activation triggers PlayAndRecord-like AVAudioSession, defaulting to earpiece. No programmatic override available from web.
 Limited
Some control is possible but not comprehensive. Cannot reliably switch between earpiece and speaker.
setSinkId() technically supported but mobile Chrome doesn’t fully expose output device enumeration. No reliable earpiece/speaker toggle from web.
 Full control
Speaker, earpiece, or Bluetooth, independently of what else is happening.
AVAudioSession: categories, overrideOutputAudioPort(.speaker), defaultToSpeaker, Bluetooth routing options, preferred input/output selection.
 Full control
Full control: speakerphone, earpiece, Bluetooth, per-stream routing.
AudioManager: setSpeakerphoneOn(), setBluetoothScoOn(), device enumeration, AudioAttributes for per-stream routing.
Volume control
app-controlled audio levels, show-driven mixing
 Locked to hardware buttons
Cannot be controlled from the browser. Always reflects the physical button setting. Can attenuate within the page's own audio mix, but cannot touch system volume.
HTMLMediaElement.volume always reads 1.0, setter ignored. GainNode in Web Audio API works within the processing graph only. System/media volume immutable from web.
 In-page audio only
Can control volume for audio within the page. Cannot control system volume.
HTMLMediaElement.volume (0.0–1.0) settable. GainNode works. System stream volumes inaccessible from web.
 Limited
Apps can show a volume slider. Apple discourages direct programmatic control of system volume.
MPVolumeView provides system volume slider UI. Direct volume manipulation via deprecated APIs. AVAudioSession output volume read-only.
 Full control
Full control over any audio stream independently.
AudioManager.setStreamVolume() for STREAM_MUSIC, STREAM_RING, STREAM_ALARM, STREAM_NOTIFICATION, STREAM_VOICE_CALL.

Push notifications
pre-show reminders, real-time game alerts, follow-up messages
 Not from a browser tab
Not available from a browser tab. Works if the experience is saved to the home screen.
No Web Push from browser tab. iOS PWA (Home Screen, display:standalone) supports Web Push since iOS 16.4. Apple briefly planned to remove PWA support in the EU under the DMA but reversed that decision before iOS 17.4 shipped.
 Yes
Works from any web page, no install needed. Requires user permission. Supports images, action buttons, and silent background updates.
Web Push API since Chrome 42. Rich notifications, silent/data-only push via FCM, background delivery through Service Workers.
 Yes
Images, action buttons, Live Activities on the lock screen, critical alerts that cut through Do Not Disturb.
APNs: 4 KB payload, rich attachments via Notification Service Extension, Live Activities, Time Sensitive / Critical Alerts, provisional notifications.
 Yes
Full notification system. Queued for up to 28 days if the device is offline.
FCM: notification channels, topic messaging, silent data messages, 28-day offline delivery queue.
Bluetooth
headphones, physical controllers, BLE beacons, wearables
 No
Blocked. Apple has explicitly declined to implement this, and every browser on iOS uses Apple's engine, so there is no workaround.
Web Bluetooth API not implemented in WebKit. Apple explicitly declined in 2020, citing fingerprinting concerns. All iOS browsers use WebKit; no alternative engine permitted.
 Can connect to devices; can’t be one
Can connect to Bluetooth devices. Cannot advertise itself as one. Requires a user gesture to initiate, and the user must pick from a system chooser each time.
Web Bluetooth API since Chrome 56: GATT central role, device discovery with filters, characteristic read/write/notifications. No peripheral/advertising, no background scanning, no Bluetooth Classic.
 Full access
Connect to devices or act as one. Background scanning. Full profile support.
CoreBluetooth: central + peripheral roles, background BLE, full GATT client/server, L2CAP channels, iBeacon. Bluetooth Classic via ExternalAccessory (MFi required).
 Full access
Connect to devices or act as one. Background scanning. Full Bluetooth Classic and BLE.
android.bluetooth: central + peripheral, background scanning, BLE advertising, Bluetooth Classic — RFCOMM, A2DP, HFP, L2CAP CoC.
NFC
tap-to-join, physical tickets, contactless triggers
 No
Blocked. Apple blocked this alongside Bluetooth in 2020.
Web NFC not implemented in WebKit. Blocked by Apple in 2020 alongside Web Bluetooth.
 Read and write tags only
Can read and write NFC tags. Cannot emulate a card (e.g. transit payment). Requires a user gesture to start scanning.
Web NFC (NDEFReader) since Chrome 89: NDEF read/write/makeReadOnly. No low-level tag access, no Host Card Emulation, foreground only.
 Read and write tags
Read and write a wide range of tag types. No card emulation (Apple Pay uses a separate chip).
CoreNFC (iPhone 7+): NDEF read/write (iOS 13+), background scanning (iOS 12+), ISO 7816, ISO 15693, FeliCa, MIFARE. No HCE.
 Read, write, and emulate
Full NFC including card emulation. The phone can act as an NFC card, not just a reader.
Full NDEF, low-level tag protocols (NfcA/B/F/V, IsoDep, MifareClassic/Ultralight), Host Card Emulation (HCE, API 19+), reader mode, background tag dispatch.
Live video & audio streaming
real-time camera feeds, peer-to-peer audio, video calling
 Works, with caveats
Works. Requires camera and/or microphone permission. Cannot share the phone's screen. Streams pause if the user switches to another app.
WebRTC supported since Safari 11. H.264 prioritized; VP8 added in Safari 12.1 (software-decoded); VP9 decode-only since Safari 17.4 (no encode in WebRTC); no AV1 in WebRTC. getDisplayMedia() unsupported on iPhone. Streams pause when Safari backgrounded. Limited simulcast (since 12.1, less capable than Chrome/Firefox). No Insertable Streams.
 Full
Works fully. Requires camera and/or microphone permission. Screen sharing available. Streams survive briefly when backgrounded.
Full WebRTC: VP8, VP9/SVC, H.264, AV1, simulcast, getDisplayMedia(), Insertable Streams. Streams survive backgrounding with limits.
 Full
All codecs. Background operation. Integrates with the phone's call system.
libWebRTC: all codecs, background operation, CallKit integration, custom processing pipelines.
 Full
All codecs. Background operation. Integrates with Android's call system.
libWebRTC: all codecs, ConnectionService integration, background operation.
Keeping connections alive
maintaining live game state, staying connected during a show
 Drops within ~30 seconds
Connections drop within about 30 seconds if the user switches apps, even briefly. Applies to home-screen web apps too. A real risk for live show scenarios.
Safari aggressively suspends background tabs and PWAs within ~30s. WebSocket connections drop. No Background Sync API. Service Workers have strict time limits.
 More forgiving
More lenient. Connections survive longer when backgrounded. Installed web apps can do limited background work.
More lenient backgrounding; connections may survive several minutes before Doze mode. Background Sync (Chrome 49+), Periodic Background Sync and Background Fetch for installed PWAs.
 Yes
Connections and background tasks stay alive. Audio, location, and call modes can run indefinitely.
BGTaskScheduler: BGAppRefreshTask (30s), BGProcessingTask. Background modes: audio, location, VoIP, BLE. URLSession background transfers.
 Yes
Persistent connections, background tasks, foreground services for long-running work.
WorkManager, foreground services with persistent notification, AlarmManager, FCM-triggered background work.
Distribution
Getting there without a full install
scan-to-play, tap-to-join, no App Store friction
 App Clips
Scan a QR code or tap an NFC tag and a small native app loads instantly, no App Store or install needed. 15 MB limit when triggered physically (QR, NFC); 50 MB when triggered digitally (Safari banner, Messages, Maps link). Full native capabilities: haptics, AR, Bluetooth, accurate sensors, flashlight. Deletes itself after about a month.
App Clips (iOS 14+): 15 MB limit for physical invocations (NFC, QR, App Clip Codes); 50 MB for digital (Safari Smart App Banner, Messages, Maps). Full native APIs available: camera, ARKit, Core Haptics, CoreBluetooth (foreground), NFC read, Apple Pay, Metal, Sign in with Apple. Exclusions: HealthKit, HomeKit, CallKit, SensorKit, background processing, full push notifications. Ephemeral push for 8 hours post-use. Auto-deleted after ~30 days. Requires companion full app in App Store.
 No equivalent
Android had an equivalent (Instant Apps) but Google discontinued it in late 2025. A web page or home-screen web app is the current lightweight option.
Android Instant Apps discontinued December 2025. Play Instant APIs no longer functional. PWA is the current recommended lightweight alternative.
