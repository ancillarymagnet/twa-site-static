Feel & feedback
Vibration & haptics
button taps, score pulses, collision rumble
 No
Apple blocks this entirely in the browser.
navigator.vibrate() not implemented in WebKit. No path to Taptic Engine from the browser. No workaround exists.
 Basic only
Android phones can vibrate from a browser, but it is blunt: on/off and simple patterns only. Requires a user gesture (tap or click) to trigger.
navigator.vibrate() supported — on/off and pattern arrays. No amplitude control, no frequency shaping, no semantic effect types.
 Full
Full access to the Taptic Engine. Every tap, click, and pulse style. Can sync with audio.
Core Haptics (CHHapticEngine, iOS 13+): transients, continuous events, intensity/sharpness 0–1, parameter curves, AHAP patterns, synchronized audio+haptics.
 Full
Full control over strength, rhythm, and type. However, the same haptic command can feel completely different across phone brands due to hardware variation (different actuator types and firmware).
VibrationEffect with amplitude control (API 26+); VibrationEffect.Composition with named primitives — PRIMITIVE_CLICK, PRIMITIVE_LOW_TICK, etc. (API 30+). Major OEM fragmentation: LRA vs ERM actuators, different HAL implementations. No amplitude interpolation — smooth ramps require many small steps. Cannot sync haptics with audio through the vibrator API (unlike CoreHaptics).

Sensors
Tilt & rotation
steering, aiming, physical gesture controls
 Drifts over time
Tilt and rotation work, but compass direction drifts over time without correction. Fine for simple tilt controls, not reliable for pointing or aiming. Requires a permission prompt that can only be shown after a user gesture (tap or click). If the user denies it, you cannot ask again.
DeviceOrientationEvent: accel + gyro only. Alpha (yaw) is relative, no magnetometer fusion, drifts over time. absolute always false. Requires DeviceOrientationEvent.requestPermission() (iOS 13+); if denied, origin cannot re-prompt.
 Accurate and stable
Android browsers combine accelerometer, gyroscope, and compass into a stable, drift-free reading. Direction stays accurate. No permission prompt needed.
Generic Sensor API (Chrome 67+): AbsoluteOrientationSensor fuses accel + gyro + magnetometer into drift-corrected quaternion. No permission prompt. ~60 Hz cap.
 Precise and fast
All sensors fused together. Updates 100+ times per second.
CMDeviceMotion (CoreMotion): full AHRS, quaternion output, 4 reference frames, bias-removed rotation rate, 100+ Hz configurable.
 Precise and fast
Same as iPhone app. Updates hundreds of times per second on high-end hardware.
TYPE_ROTATION_VECTOR (full fusion), TYPE_GAME_ROTATION_VECTOR, quaternion output, up to 200–500+ Hz. HIGH_SAMPLING_RATE_SENSORS permission above 200 Hz (Android 12+).

Sensor latency
motion controls, real-time input, timing-sensitive mechanics
 High
Sensor events go through the browser’s main thread and are subject to scheduling delays. Expect 50–100ms of latency in practice. Fine for slow gestures, noticeable for anything timing-sensitive.
DeviceMotionEvent delivered via main thread event loop. Subject to JS scheduling jitter and browser throttling. Typical observed latency 50–100ms. No way to request lower latency or higher priority from web.
 High
Same main-thread bottleneck as iPhone browser. Chrome’s Generic Sensor API has slightly better architecture but is still meaningfully slower than native.
Generic Sensor API uses a dedicated sensor task runner but data still crosses to the main thread before JS callbacks fire. Latency improved over DeviceMotionEvent but still 30–80ms in practice.
 Low
Reads directly from hardware with no browser in the way. Typical latency under 5ms. The difference is perceptible for fast motion or precise timing.
CoreMotion delivers data on a dedicated background thread at hardware interrupt frequency. Typical latency 1–5ms. No main-thread involvement until the app chooses to process.
 Low
Same as iPhone app. Direct hardware access, sub-5ms latency.
SensorManager callbacks on a dedicated SensorThread. Typical latency 1–5ms at SENSOR_DELAY_FASTEST. No main-thread bottleneck.

overlaying graphics on the real world, spatial experiences
 Limited AR
No WebXR, so no surface detection or world tracking. Marker-based and image-based AR work via JS computer vision libraries (MindAR, AR.js). 8th Wall shut down in early 2026. AR Quick Look exists but is non-programmable and broken in in-app browsers (Facebook, Instagram, WeChat). WebGPU in Safari 26 (Fall 2025) may improve JS-based AR performance.
No WebXR support in Safari (any version, including Safari 26). A WebXR AR Module feature flag exists but does nothing. ARKit: world/face/body tracking, LiDAR scene reconstruction, image/object detection, people occlusion, location anchors, 4K/HDR AR video.

Face, hand & body tracking
gesture controls, face filters, conductor input, physical performance
 Possible, but slower
Works via AI models running in the page (TensorFlow.js/MediaPipe). Covers face mesh, hand landmarks, and body pose. Frame rates are lower and battery drain is higher than a native app. TensorFlow.js runtime actually outperforms MediaPipe’s own runtime on iOS specifically. WebGPU (Safari 26) may further narrow the gap with native.
MediaPipe (WebAssembly + WebGL): Face Mesh (468 points), Hands (21 landmarks/hand), Pose (33 landmarks). ~20–30 FPS for individual tasks; holistic (all three) significantly slower. TF.js runtime faster than MediaPipe runtime on iOS (counter-intuitively). MediaPipe has known WebGL framebuffer bugs on iOS Safari. No Neural Engine access. WebGPU (Safari 26, Fall 2025) could improve inference performance.
 Possible, better than iPhone browser
Same libraries as iPhone browser, but typically 30–60+ FPS with GPU delegation enabled. Chrome’s WebGL and WASM implementations are better optimized for this workload.
Same MediaPipe stack. With GPU delegation: 60+ FPS (vs ~10–15 FPS CPU-only WASM). Chrome’s WebGL/WASM significantly outperforms iOS Safari for ML inference. Still no NNAPI access from the browser.
 Fast and precise
Hardware-accelerated, built into the OS. Face mesh, 21-point hand tracking, full body skeleton. About 10ms response time.
Vision framework: VNDetectFaceLandmarksRequest (76 landmarks), VNDetectHumanHandPoseRequest (21 landmarks, 2 hands), VNDetectHumanBodyPoseRequest (19 joints). ARKit face/body tracking on A12+. ~5–14 ms latency.
 Fast and precise
Uses the phone's AI chip. Face mesh, hand, and body tracking at 2-3x faster than the browser equivalent.
ML Kit Face Mesh Detection (468 landmarks), Pose Detection (33 landmarks), GPU/NNAPI acceleration. ~2–3× lower latency than browser implementations.
Audio
Playing sound automatically
ambient music on load, sound effects timed to show events
 Can’t start without a tap
No audio until the user taps something first. Cannot start audio on page load or on a timer. Hidden gotcha: the hardware mute/ringer switch silences Web Audio API output but does not silence <audio> elements — the two paths behave differently. Libraries like unmute-ios-audio work around this by keeping a silent <audio> element playing. Preloading audio data (decoding buffers) is allowed; only playback initiation is gated.
WebKit autoplay policy: any audible media (HTMLMediaElement, AudioContext) blocked until user gesture. AudioContext starts suspended; must resume() after interaction. No per-site exemptions. Applies equally to iOS PWAs (no installed-app exemption). Mute switch gotcha: Web Audio API honours the ringer switch (silences output); HTMLMediaElement does not (plays through speaker regardless). WebKit bug #237322. Workaround: play a silent <audio> element in parallel to force Web Audio onto the media channel.
 Needs a tap first — unless installed
Same restriction in the browser tab. If installed to the home screen as a PWA, the autoplay requirement goes away entirely — audio can play on load without any user gesture. No hardware mute switch issue like iOS.
Chrome autoplay policy: same gesture requirement in browser tab. Installed Android PWA exempt from autoplay restriction since Chrome 71 — AudioContext can resume() on load. No mute switch discrepancy; volume is unified through system streams. Chrome MEI (Media Engagement Index) may grant autoplay on desktop but is less relevant on mobile.
 Play whenever
No restrictions. Sound plays when you tell it to.
AVAudioSession with .playback category. No gesture requirement.
 Play whenever
No restrictions. Sound plays when you tell it to.
MediaPlayer, ExoPlayer, AudioTrack — no autoplay restrictions.

Nearby Interaction framework: U1 chip (iPhone 11/12/13/14), U2 chip (iPhone 15/16 — 7 nm, more power-efficient). No UWB on iPhone SE (any gen) or iPhone 16e. NINearbyObject with distance + direction, background sessions (iOS 16+), third-party accessory sessions via NI Accessory Protocol. No cross-platform phone-to-phone ranging — Apple’s session-token exchange is proprietary. Requires NSNearbyInteractionUsageDescription; permission is one-time-per-app-launch (resets on quit). Ultrasonic communication
phone-to-phone data over speaker & mic, cross-device pairing without internet
 No
Blocked in practice. Web Audio API and microphone access exist, but the iPhone’s hardware anti-aliasing filter rolls off above ~20 kHz and the default 48 kHz sample rate caps capture at 24 kHz (Nyquist limit). Even within that range, iOS audio processing attenuates near-ultrasonic content enough to make reliable data transfer impractical. No background mic access.
getUserMedia + Web Audio API are available, but iPhone MEMS mic ADC anti-aliasing rolls off between 20–24 kHz. Default sample rate is 48 kHz (Nyquist: 24 kHz). iOS audio pipeline processing further attenuates near-ultrasonic frequencies. No background audio capture. Libraries like ggwave have an ultrasonic mode (F0 = 15 kHz, channel width up to 4.5 kHz), but reception is unreliable on iOS Safari.
 Possible with effort
Can generate and detect near-ultrasonic tones (~15–20 kHz) using the Web Audio API and microphone access. Android hardware is more permissive with these frequencies than iOS. Still low-bandwidth (~8–16 bytes/sec), short-range, and foreground-only. Requires mic permission, which triggers a visible prompt.
Web Audio API (AudioContext + AnalyserNode) for tone generation; getUserMedia at 48 kHz for capture. Android doesn’t aggressively filter near-ultrasonic frequencies. Libraries like ggwave work in-browser via WASM (ultrasonic protocols at F0 = 15 kHz, FSK modulation, Reed-Solomon ECC, 8–16 bytes/sec). Foreground-only; no background mic access. Requires user gesture to start AudioContext on first interaction.
 Yes, with caveats
Full control over audio hardware. Can generate and capture near-ultrasonic frequencies and run in background audio mode. However, the same 48 kHz default sample rate and hardware anti-aliasing apply — you’re working in the 15–22 kHz band, not true ultrasound. Bandwidth tops out around 8–16 bytes/sec. The main commercial SDK (Chirp) was acquired by Sonos in 2020 and its public SDK shut down; ggwave is the primary open-source alternative.
AVAudioEngine / AudioUnit: default 48 kHz sample rate (Nyquist: 24 kHz). iPhone MEMS mic anti-aliasing rolls off 20–24 kHz. Background audio mode available. Chirp SDK discontinued (acquired by Sonos, Feb 2020; SDK removed Mar 2020). ggwave (MIT, actively maintained): FSK modulation, Reed-Solomon ECC, ultrasonic protocols at F0 = 15 kHz, 8–16 bytes/sec, Swift package available. SoniTalk (open protocol, academic) is another option.
 Yes, with caveats
Full control over audio hardware. Can capture near-ultrasonic frequencies and run via foreground service in the background. Same bandwidth and range limits as iOS — this is fundamentally a low-bandwidth, short-range channel (~8–16 bytes/sec, a few metres). Works cross-platform with iOS devices since it’s just sound. Chirp SDK shut down; ggwave is the main open-source option.

Chirp SDK discontinued (Sonos acquisition, 2020). ggwave (MIT): FSK modulation, Reed-Solomon ECC, ultrasonic F0 = 15 kHz, 8–16 bytes/sec. Getting there without a full install

 App Clips
Scan a QR code or tap an NFC tag and a small native app loads instantly, no App Store or install needed. 15 MB limit when triggered physically (QR, NFC); 50 MB when triggered digitally (Safari banner, Messages, Maps link). Full native capabilities: haptics, AR, Bluetooth, accurate sensors, flashlight. Deletes itself after about a month.
App Clips (iOS 14+): 15 MB limit for physical invocations (NFC, QR, App Clip Codes); 50 MB for digital (Safari Smart App Banner, Messages, Maps). NFC tags can trigger App Clip cards even with screen locked. Full native APIs available: camera, ARKit, Core Haptics, CoreBluetooth (foreground), NFC read, Apple Pay, Metal, Sign in with Apple. Exclusions: HealthKit, HomeKit, CallKit, SensorKit, background processing, full push notifications. Ephemeral push for 8 hours post-use. Auto-deleted after ~30 days. For PWAs: no beforeinstallprompt API — users must manually use Share > Add to Home Screen. iOS 26 defaults “Open as Web App” toggle on for all Home Screen additions.
 No equivalent
Android had an equivalent (Instant Apps) but Google discontinued it in late 2025. A web page or home-screen web app is the current lightweight option. Android Chrome supports beforeinstallprompt, so you can programmatically prompt users to install — unlike iOS, which has no install prompt API.
Android Instant Apps discontinued December 2025. Play Instant APIs no longer functional. PWA is the recommended lightweight alternative.
